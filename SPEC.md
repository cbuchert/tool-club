## Product premise

Tool Club is a private, invite-only community for a small group of friends and
makers in Salt Lake City. Members attend events, propose ideas, and document
what happened. The app handles logistics (RSVPs, headcounts, calendars) and memory
(recaps, photos). It does not try to be a social network or a forum.

Membership is small by design — tens of members, not hundreds.

## Membership and access

### Roles

**Anonymous**: Can view the public landing page and public RSS feed. Can use a valid
invite link to begin account creation.

**Member**: A verified, active user. Can access all member features described below.

**Admin**: A member with elevated privileges. Admins are designated directly in the
database. There is no UI for self-service role elevation. Multiple admins are
supported.

**Host (contextual)**: Any member designated as the host of a specific event. Host
privileges are scoped to that event only — writing a recap, toggling photo visibility.

### Invite system

- Any active member can generate an invite link.
- Invite links are single-use and expire after 30 days.
- A member may have at most one pending (unused, unexpired) invite at a time.
  Generating a second invite requires revoking the first.
- Invite links carry a random token: `toolclub.app/join/[token]`
- The invite records who was invited (optional email, for tracking) and who invited them.
- Admins can see all pending invites and revoke any of them.
- When an invite is revoked or expires, the row is hard-deleted.
- The invite chain (who invited whom) is preserved in `users.invited_by` at the time
  of account creation, even if the inviter later deletes their account.

### Account creation (invite flow)

1. Prospective member visits an invite link.
2. App validates the token: exists, not expired, not redeemed.
   - If invalid: show an error page with the specific reason (expired vs. redeemed)
     and the name of the person who issued the invite.
3. Prospective member enters their email and display name.
4. Supabase Auth sends a magic link to that email.
5. Prospective member clicks the link, lands at `/auth/callback`.
6. App creates a `users` row with `role = 'member'`, marks the invite as redeemed.
7. Member is redirected to the events list.

### Sign in (existing members)

1. Member enters their email on the sign-in page.
2. Supabase Auth sends a magic link.
3. Member clicks the link, session is established.
4. Member is redirected to the events list.

If an email that has no account attempts sign-in (not the invite flow), they receive
the magic link email but land on an error page explaining that they need an invite.
Do not reveal whether an email is registered to an anonymous visitor.

### Account deletion

When a member deletes their account:

- `users` row is anonymized: `display_name` set to "Former member", `email` and
  `avatar_url` nulled out, row is retained.
- All content (events hosted, suggestions, comments, recaps) remains attributed to
  the anonymized record.
- All pending invites generated by this member are revoked (hard-deleted).
- The `invited_by` reference on members they recruited is preserved.
- Feed tokens are deleted (their RSS/iCal feeds immediately stop working).

**Blocker**: A member cannot delete their account if they are the host of any
upcoming (non-past) event. The UI surfaces this with the event name and instructs
them to ask an admin to reassign the host.

### Suspension

Admins can suspend a member. A suspended member:

- Cannot sign in (Supabase Auth account is disabled).
- Their feed tokens are invalidated.
- Their content remains visible.
- Can be reinstated by an admin.

## Events

### States

Events move through states in one direction only:

```
draft → published → past
```

- **Draft**: Visible only to admins. Not indexed in feeds.
- **Published**: Visible to all members. Appears in RSS/iCal feeds.
- **Past**: Set automatically by a cron job. RSVPs are locked.

There is no "cancelled" state in v1. Admins can delete a published event (hard delete).
Members who RSVPed are not automatically notified — notification is out of scope for v1.

### Event fields

| Field             | Notes                                                                                         |
| ----------------- | --------------------------------------------------------------------------------------------- |
| Title             | Required                                                                                      |
| Date + start time | Required                                                                                      |
| End time          | Optional                                                                                      |
| Host              | Required. A member name. Not a foreign key to users — freeform text to allow external hosts.  |
| Host user         | Optional FK to users. Set when the host is a member. Enables the "host can write recap" rule. |
| Venue name        | Optional                                                                                      |
| Address           | Optional                                                                                      |
| Capacity          | Optional. If set, RSVPs are capped at this number.                                            |
| Body              | Markdown. Rendered on the event detail page.                                                  |
| Links             | A list of label+URL pairs. Optional.                                                          |
| Status            | draft / published / past                                                                      |
| Promoted from     | Optional FK to suggestions. Set when an event is created from a suggestion.                   |

### Event list display

- Events are grouped by month/year (Denver time) with a section header (e.g. "April 2026").
- Within each month, events are ordered by date ascending.
- Past events are ordered most-recent-first within their month groups.

### Event detail display

- The address field renders as a Google Maps hyperlink (`maps.google.com/?q=…`).

### RSVPs

- Members RSVP "Going" or "Can't make it."
- Members can change their RSVP any time before the event starts.
- Once the event is `past`, RSVPs are locked and no changes are accepted.
- "Going" RSVPs are visible to all members on the event detail page.
- "Can't make it" RSVPs are visible only to admins and the RSVP owner.
- If an event has a capacity set: once the "going" count reaches capacity,
  the "Going" button is disabled for members who have not already RSVPed yes.
  A message directs them to contact the host.

### Recaps

- A recap can be written by the designated host user of an event (if they are a member)
  or by any admin.
- Only one recap per event.
- Recaps are written in Markdown.
- Photos can be attached by any member after the recap is published.
- The event host (or admin) can toggle each photo between `public` and `private`.
  - Public photos appear in the public RSS feed.
  - Private photos are visible to members only.
- There is no approval step for photo uploads. Any member can upload.
- Photo uploads go to Supabase Storage server-side.

## Suggestions

### Purpose

Members propose event ideas. The group votes. Admins promote top suggestions to events.

### States

```
open → planned
     ↘ closed
```

- **Open**: Voting is active. Any member can vote or comment.
- **Planned**: Admin has promoted this suggestion to an event. Voting is closed.
  A link to the resulting event is shown.
- **Closed**: Admin has closed voting without promoting. No further action.

### Fields

| Field             | Notes                                                 |
| ----------------- | ----------------------------------------------------- |
| Title             | Required                                              |
| Body              | Markdown. The pitch. Required.                        |
| Nominated host    | Freeform text. Who the proposer thinks should run it. |
| Voting closes at  | A date. After this date, voting is locked.            |
| Status            | open / planned / closed                               |
| Promoted to event | FK to events. Set when status becomes planned.        |

### Voting

- One vote per member per suggestion.
- Members can un-vote (toggle).
- Votes are visible as a count only — individual votes are not exposed to members.
  (Admins can see individual votes.)
- Voting is locked when `voting_closes_at` has passed or status is not `open`.

### Comments

- Any member can comment on any open or planned suggestion.
- Comments are plain text (no Markdown).
- Members can delete their own comments.
- Admins can delete any comment.
- No editing. Delete and re-post.

### List display

The suggestions list renders three sections in order:

1. **Open** — always visible; sorted newest first; labelled "N open · newest first".
2. **Planned** — always visible; labelled "N planned"; each card includes a
   "→ view event" link to the resulting event.
3. **Closed** — visible by default; labelled "N closed" with a "Hide ↑ / Show ↓"
   toggle. Closed suggestions are shown rather than hidden because they represent
   institutional memory — members may want to reference why an idea was shelved.

### Promote to event

When an admin promotes a suggestion to an event:

1. A new event is created in `draft` status, pre-filled from the suggestion fields.
2. `suggestions.status` is set to `planned`, `suggestions.promoted_to_event` is set.
3. The suggestion detail shows a banner linking to the event.
4. The admin is taken to the event edit screen to complete and publish the event.

## Feed and calendar

### Private feeds (per-member)

Each member has a unique feed token. Feeds are authenticated by token in the URL.

- **RSS**: All published events (titles, dates, locations, body), published recaps.
- **iCal**: All published events as calendar entries with dates, times, locations.

Tokens are not JWTs. They are random strings stored in `feed_tokens`. Token
regeneration immediately invalidates existing subscriptions.

### Public feed

- **RSS only**: Published event titles and dates. No locations, no member names,
  no addresses, no recaps.
- No authentication required.
- URL: `/feed/public`

## Admin capabilities

Admins have full read/write access to all content. Specific capabilities:

**Events**: Create, edit, publish, delete any event. Set event to `past` manually
if needed. Reassign host.

**Members**: View all members and their invite history. Suspend/reinstate members.
View all pending invites. Revoke any invite. Generate invite links (same rules as
members: one pending at a time per admin account).

**Suggestions**: Change status (close, re-open, promote to event). Delete any comment.

**Recaps**: Write or edit a recap on any event (not just events they hosted).

**Headcount**: View full RSVP list for any event, including "can't make it" responses.

## Account settings

### Display name

Members can edit their display name inline. Changes are immediately reflected across
the app. No character limit is enforced in v1, but the UI input suggests reasonable
length.

### Avatar

Members can upload a profile photo. Stored in Supabase Storage. If no photo is
uploaded, the avatar falls back to the first letter of the display name on a colored
background. The color is deterministic based on user ID (not random on each render).

### Invites

The account page shows:

- A button to generate an invite link (disabled if one is pending).
- The member's invite history: who joined via their link, and any pending invite
  with its expiry date and a revoke action.
- The member's "network": members they invited and members those members invited
  (one level deep only).

### Feed and calendar

Described above. Token regeneration is a destructive action and requires a
confirmation step that explains what breaks.

## Out of scope for v1

The following are explicitly deferred. Do not design the v1 schema or routes to
accommodate them unless they fall out naturally.

- Email notifications (new event, RSVP reminders, recap posted)
- Waitlists for capacity-capped events
- Rich text editor for recaps or suggestions (Markdown input only)
- Member-to-member messaging
- Event search or filtering beyond upcoming/past toggle
- Mobile push notifications
- Photo ordering or galleries beyond a simple grid
- Multiple admins managing concurrent drafts (no conflict resolution in v1)
